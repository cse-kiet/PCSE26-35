## Key design decisions

### Why is `run_sumo_sim` a separate file?

In MATLAB, a function defined as a "local function" inside a script (using the
`function` keyword at the bottom) cannot be called from another script file.

`run_trials.m` and `stress_test.m` both need to call the simulation engine.
If `run_sumo_sim` were a local function inside `intersectionmultipolicy_sumo.m`,
it would be invisible to the other two scripts.

Solution: `run_sumo_sim.m` lives in `src/` as a standalone function file.
Any script that does `addpath('src')` can call it directly.

### Why are policies separate files too?

Same reason. The policy functions are called by `run_sumo_sim`, which is itself
called from three different top-level scripts. Keeping them as separate `.m` files
in `src/` means they are accessible from everywhere with one `addpath('src')` call.

The policy signatures were also updated — they now take `BANDWIDTH` as an explicit
argument instead of reading it from a MATLAB `global` variable. This makes each policy
self-contained and easier to test in isolation.

### Why does MATLAB call `addpath('src')` instead of putting scripts in `src/`?

The top-level scripts (`intersectionmultipolicy_sumo.m`, `run_trials.m`,
`stress_test.m`) are meant to be run from the project root — the same place as the
original `intersection3.m`. A user can just open MATLAB, `cd` to the project folder,
and type the script name. No subdirectory navigation required.

The path `src/sumo/fcd_traces.csv` is relative to the project root, so this only
works correctly when MATLAB's working directory is the project root.

### Why is SUMO run offline (not with TraCI)?

TraCI is SUMO's real-time control interface — it lets Python/MATLAB step SUMO one
timestep at a time and query vehicle states mid-simulation. It is powerful but complex.

For this project, the mobility and the offloading logic are independent:
- The vehicle positions don't depend on offloading decisions
- The offloading decisions only read positions, they don't affect movement

Offline FCD parsing is simpler, faster, and fully reproducible (same CSV = same results).
TraCI would add latency, inter-process communication, and synchronisation complexity
for no scientific benefit.

### Why does `channel_model.m` use deterministic shadowing?

Real shadowing is a random Gaussian process: `X ~ N(0, sigma^2)`. If we sample it
fresh at every timestep, the link quality between two vehicles fluctuates randomly
even when they are standing still — which would be physically wrong.

In a real system, shadowing is caused by static objects (buildings, trucks) and
changes slowly as vehicles move. A common approximation for simulation is to use
a spatially-correlated random field seeded by vehicle position.

Here we use `sigma * sin(d * 0.1)` — a deterministic function of distance that
approximates the oscillatory nature of shadowing without adding randomness.
This keeps the simulation reproducible and avoids link quality jitter.
